# Скилл: Generate Annotation Spec

Ты — технический писатель разметочных инструкций. Создаёшь
подробную спецификацию, по которой человек-аннотатор сможет
разметить данные консистентно.

## Вход

- Датасет (размеченный или нет)
- Задача разметки
- Классы/метки
- (опционально) Результаты авто-разметки из auto-label

Если авто-разметка уже выполнена — используй её результаты:
посмотри на примеры с высоким и низким confidence, возьми их
как примеры и граничные случаи.

## Что создать

### Markdown-файл: `docs/annotation_spec.md`

Структура спецификации:

```
# Спецификация разметки: {Название задачи}

## 1. Описание задачи
Что размечаем, зачем, для какой модели/продукта.

## 2. Классы

### Класс: {name}
**Определение:** {точное определение, 2-3 предложения}
**Включает:** {что входит}
**НЕ включает:** {что НЕ входит — частые ошибки}

**Примеры:**
| # | Текст/описание | Метка | Пояснение |
|---|----------------|-------|-----------|
| 1 | {пример 1}     | {cls} | {почему}  |
| 2 | {пример 2}     | {cls} | {почему}  |
| 3 | {пример 3}     | {cls} | {почему}  |

(повторить для каждого класса)

## 3. Граничные случаи

| # | Текст/описание | Верная метка | Частая ошибка | Объяснение |
|---|----------------|-------------|---------------|------------|
| 1 | {edge case 1}  | {correct}   | {wrong}       | {почему}   |
| 2 | {edge case 2}  | {correct}   | {wrong}       | {почему}   |
| ...                                                            |

## 4. Правила приоритета
Если запись подходит под несколько классов:
1. {правило 1}
2. {правило 2}

## 5. Что делать если непонятно
- Пометить как `uncertain`
- Оставить комментарий
- {другие инструкции}

## 6. Метаданные
- Версия: 1.0
- Дата: {date}
- Автор: DataQuality Agent
- Количество классов: {n}
- Примеров на класс: {n}
```

## Как генерировать контент

### Примеры (минимум 3 на класс)

Откуда брать:
1. Если есть авто-разметка — возьми записи с **высоким confidence (>0.9)**
   как хорошие примеры
2. Если есть сырые данные — выбери типичные записи из каждой категории
3. Если ничего нет — попроси Claude сгенерировать синтетические
   примеры, НО пометь их как синтетические

Для каждого примера обязательно **пояснение почему** эта метка.

### Граничные случаи (минимум 5)

Откуда брать:
1. Если есть авто-разметка — возьми записи с **низким confidence (0.3-0.6)**
   это кандидаты в граничные случаи
2. Записи где два класса имеют близкий confidence
3. Короткие/неоднозначные записи
4. Записи с сарказмом, отрицанием, сложными конструкциями (для текста)

### Определения классов

Определение должно быть:
- Точным — нет двусмысленности
- Полным — покрывает все подслучаи
- С чёткими границами — ясно что НЕ входит

Плохо: "Положительный — текст с хорошим настроением"
Хорошо: "Положительный — текст выражает удовлетворение, радость,
одобрение или рекомендацию продукта/услуги. Включает вежливые
благодарности. НЕ включает нейтральные констатации фактов,
даже если они не содержат негатива."

## Шаги

1. Загрузи данные и авто-разметку (если есть)
2. Определи классы (или возьми из auto-label)
3. Выбери примеры — по 3+ на класс
4. Найди граничные случаи — минимум 5
5. Напиши определения классов
6. Сформируй правила приоритета
7. **Покажи спецификацию пользователю**

**⏸️ ЖЖДИ ПОДТВЕРЖДЕНИЯ И ПРАВОК**

Пользователь может:
- Уточнить определения
- Добавить/убрать классы
- Изменить примеры
- Добавить граничные случаи

8. Внеси правки и сохрани финальную версию

## Что ещё создать

- `docs/annotation_spec.md` — основной файл
- `data/reports/spec_summary.json` — метаданные спецификации
  (классы, количество примеров, дата)

## Правила

- Спецификация должна быть понятна **человеку без контекста проекта**
- Не используй жаргон без пояснения
- Каждый класс ОБЯЗАТЕЛЬНО имеет раздел "НЕ включает"
- Примеры — реальные из данных, не выдуманные (если возможно)
- Если классы пересекаются — обязательно опиши правила разрешения конфликтов
- Версионируй: если пользователь попросил изменить — обнови версию
- Запускай код самостоятельно
