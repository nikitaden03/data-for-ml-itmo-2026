# Пайплайн: Data Quality Agent

Оркестрирующий скилл. Последовательно выполняет detect → fix → compare
с паузами для подтверждения пользователем.

## Вход

Пользователь даёт:
- Путь к CSV-файлу
- (опционально) Целевую переменную
- (опционально) ML-задачу
- (опционально) Предпочтения по стратегиям

## Выполнение

### Фаза 1: ДЕТЕКТИВ (detect-issues)

Выполни всё что описано в скилле detect-issues:
- Анализ пропусков, дубликатов, выбросов, дисбаланса
- Создай quality_utils.py
- Создай ноутбук 01_detect_issues.ipynb
- Выполни ноутбук
- Сохрани quality_report.json

Покажи результат:

```
═══════════════════════════════════════
📋 ФАЗА 1: ДЕТЕКТИВ — ЗАВЕРШЕНА
═══════════════════════════════════════

Quality Score: {score}/100

Найденные проблемы:
🔴 {critical issues}
🟠 {high issues}
🟡 {medium issues}

Ноутбук: notebooks/01_detect_issues.ipynb

Переходим к очистке?
```

**⏸️ СТОП. ЖЖДИ ОТВЕТА ПОЛЬЗОВАТЕЛЯ.**

Пользователь может:
- Сказать "да" → переходи к фазе 2
- Попросить пояснить что-то → поясни
- Сказать что делать с конкретными столбцами → учти в фазе 2
- Сказать "пропусти fix, только compare" → пропусти

### Фаза 4: ЛЕТОПИСЕЦ (readme)

Напиши `readme/README_data_quality.md` — документацию того что было сделано в этом запуске.

README должен содержать:

1. **Датасет и задача** — путь к файлу, shape, целевая переменная, тип задачи

2. **Фаза 1 — найденные проблемы** — таблица всех issues с severity, count, % и кратким описанием. Quality Score исходного датасета.

3. **Решения по каждой проблеме** — таблица: столбец → решение → обоснование.
   Если пользователь вносил правки в ходе обсуждения — отразить именно финальные решения.

4. **Фаза 2 — результаты стратегий** — описание Strategy A и B (что делает каждая),
   таблица: строки/столбцы/quality score/потеря данных для обеих.

5. **Фаза 3 — вердикт** — рекомендованная стратегия, причины выбора,
   результаты KS-тестов, рекомендации при обучении модели.

6. **Структура файлов** — дерево созданных артефактов (ноутбуки, CSV, JSON, графики).

7. **Как воспроизвести** — команды для повторного запуска ноутбуков.

Покажи результат:

═══════════════════════════════════════

## Критические правила

1. **НИКОГДА не пропускай паузы** — на каждом чекпоинте жди ответа
2. **Не перезаписывай** исходный файл — всегда работай с копиями
3. **Если шаг упал** — исправь код и повтори, не пропускай фазу
4. **Если пользователь хочет изменить стратегию** — подстройся
5. **Если пользователь хочет пропустить фазу** — пропусти, но предупреди что будет потеряно
6. **quality_utils.py должен быть единым** — все скиллы дополняют его, не дублируют
7. **Ноутбуки должны быть выполнимыми** — после создания запусти через jupyter nbconvert --execute и убедись что нет ошибок
8. **Каждый ноутбук самодостаточный** — можно открыть и понять без остальных
9. **README всегда последним** — пишется после всех фаз, отражает финальное состояние
10. **README не пересказывает код** — только решения, обоснования и результаты
